\subsection{System Description}
\label{sub:system_description}
% Begin SubSection
This system will monitor real-time sensor data input from across a city to measure environmental changes. The system will be primarily used by city operators who have access to a dashboard which shows recent data aggregated over time as well as environmental alerts. The management of alerts and accounts is done by administrators who have greater permissions than other users. The system will also expose a REST API for public users which only exposes non-sensitive environmental data aggregated by city zone. The system will also have an app for public users to easily view data from the API. All major actions are logged for audit purposes and the system implements rate-limiting to prevent denial-of-service attacks and to maintain high uptime. 
% End SubSection

\subsection{System Architecture}
\label{sub:system_architecture}
% Begin SubSection
SCHEMAS will use the Presentation-Abstraction-Control (PAC) Interaction-Oriented Software Architecture style. Our application will be structured with a hierarchy of independent agents, and each agent will have three components: Presentation (UI), Abstraction (data model), and Control (logic). \\\\
The agents are defined as follows:

\begin{table}[H]
	\centering
	\begin{tabular}{|l|l|p{8cm}|}
		\hline
		\textbf{Level} & \textbf{Agent} & \textbf{Purpose} \\
		\hline
		Top & City Agent & Central coordinator of the entire system; handles the internal dashboard for operators and administrators and orchestrates all other agents \\
		\hline
		Middle & Public Agent & Handles the public-facing REST API and public dashboard, exposing only non-sensitive, read-only information to external users \\
		\hline
		Middle & Sensor Processing Agent & Ingests and processes incoming external sensor data and distributes results to relevant agents \\
		\hline
		Bottom & Account Agent & Manages account creation, login, and account profile details for all system users \\
		\hline
		Bottom & Alert Agent & Manages alert rule creation, triggered alert views, and alert verification \\
		\hline
		Bottom & Audit Log Agent & Records system events from all agents and exposes the audit log view to authorized users \\
		\hline
	\end{tabular}
	\caption{PAC Architecture Agents}
	\label{tab:pac-agents}
\end{table}

Within the Abstractions of our architecture, there are four databases present: an account information database, an audit log database, a configured alerts database and a sensor data database (storing time series data).
\\\\
The application will have a Presentation-Abstraction-Control (PAC) Architecture Style because it structures a system as a hierarchy of cooperating and independent agents. The top-level agent will be responsible for the overall system functionality, and intermediate/bottom-level agents will handle specific tasks/responsibilities. This architecture also supports multiple views to display the same data in multiple ways simultaneously, which is extremely suited for SCEMAS since it has a City View (for internal users) and a Public View (for external users). Another reason why we decided on this architecture style is how it adheres to the separation of concerns principle. Each agent contains its own Presentation-Abstraction-Control that is responsible for its specific functionality. This can be beneficial for the system because it will be easy to maintain, replace and extend the agents when there are more business requirements without affecting the rest of the application.
\\\\
Various other architectural styles were considered for this project, but were eliminated from contention as some characteristics of them were not well-suited for the system as a whole.
\\\\
The repository architecture was considered, but eliminated due to the nature of our system’s data. Although there is a need for shared data between agents within the SCHEMAS system, the sensor data is active and thus would not work with a passive data store model.
\\\\
The blackboard architecture was eliminated due to our system not requiring specialized modules with domain expertise contributing to problem-solving. Another reason for the elimination is that our application does not have any non-deterministic problems. This would cause difficulty in deciding when to terminate reasoning.
\\\\
The Model View Controller Architecture (One and Two) was considered as an option as well, but ended up eliminating it as an option as the system requires a diverse set of data that may update at different frequencies. It was also determined that choosing the MVC architecture could negatively affect the sustainability of the system, as there is high coupling between the View and Model.

\subsection{Subsystems}
\label{sub:subsystems}
% Begin SubSection
 SCEMAS is organized into six primary subsystems: Public Access, City Dashboard, Sensor Processing, Account Management, Alert Management, and Audit Logging. These subsystems operate within the broader PAC architectural style. Each subsystem encapsulates a specific responsibility of the system while interacting with each other to support the overall functionality of the platform.

	\subsubsection*{Public Access Subsystem}
	The Public Access Subsystem manages all functionality related to external users. It provides a read-only REST API and a public dashboard that allows residents and third-party developers to access non-sensitive environmental data.

	\subsubsection*{City Dashboard Subsystem}
	The City Dashboard Subsystem supports the internal monitoring interface used by city operators and administrators. Through this subsystem, authorized users can view environmental conditions across geographic zones, monitor alerts, and analyze environmental trends. It communicates with the system's sensor data storage to get environmental measurements and works with the Alert Management Subsystem to show active alerts and related information on the dashboard.

	\subsubsection*{Sensor Processing Subsystem}
	The Sensor Processing Subsystem is responsible for ingesting and processing environmental telemetry from external sensors. Incoming data is validated, formatted, and prepared for use within the system before being stored in the sensor data repository. This subsystem sends the processed environmental data to other subsystems, specifically the City Dashboard and Alert Management Subsystems, allowing real-time monitoring and alert detection.

	\subsubsection*{Account Management Subsystem}
	The Account Management Subsystem handles user login and account management. It allows accounts to be created and ensures that access to system features is controlled using role-based access. This subsystem also works with the Audit Logging Subsystem to record important account activities such as login attempts and account updates.

	\subsubsection*{Alert Management Subsystem}
	The Alert Management Subsystem handles the creation and management of alert rules in the system. City operators can create alerts based on environmental thresholds, while administrators review and approve these alerts before they become active. When sensor data passes a defined threshold, the subsystem generates an alert and ensures it appears on the dashboard. Alert-related actions are also recorded in the Audit Logging Subsystem for tracking and auditing purposes.

	\subsubsection*{Audit Logging Subsystem}
	The Audit Logging Subsystem records important activities that occur throughout the system. It stores logs related to authentication events and alert actions. Administrators can view these logs through the audit log interface to monitor system activity and review past events.
